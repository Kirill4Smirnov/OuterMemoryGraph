# Способы хранения и обработки

Для PageRank были рассмотрены три варианта

## Разбиение по конечным вершинам

Этот вариант реализован в проекте

После внешней сортировки ребро хранится как пара `(конечная вершина, исходная вершина)`

Каждый файл отвечает за свой диапазон конечных вершин

Поток читает такой файл последовательно и записывает ранги только в свой диапазон, поэтому блокировки и атомарные сложения не нужны

В памяти остаются два массива рангов, исходящие степени и небольшие буферы

Список рёбер всё время находится на накопителе

Почему выбран этот способ

- каждое ребро читается один раз за итерацию
- чтение почти полностью последовательное
- на каждой итерации не создаются временные записи для всех рёбер
- расход памяти не зависит от степени отдельной вершины
- результат не зависит от порядка работы потоков

Главное ограничение — массивы состояния вершин должны помещаться в оперативную память

## Рассылка и сбор обновлений

Сначала обновления рассылаются по временным файлам, затем собираются по конечным вершинам

Сначала для каждого ребра на диск записывается вклад исходной вершины, затем эти записи группируются и суммируются по конечным вершинам

Ранги можно обрабатывать частями, поэтому в памяти не обязательно держать состояние всего графа

Недостаток заметный — на каждой итерации приходится записывать и снова читать `O(E)` обновлений

Для PageRank это создаёт лишнюю нагрузку на накопитель

## Двумерное разбиение

Матрица смежности делится на блоки по исходным и конечным вершинам

В памяти находятся только ранги исходного блока и накопитель конечного блока

Размер блока можно подобрать под жёсткий предел памяти, даже если полный массив рангов не помещается

За это приходится платить повторным чтением рангов и более сложным порядком обработки блоков

## Сравнение

| Способ | Память | Работа с диском за итерацию | Итог |
|---|---:|---:|---|
| Разбиение по конечным вершинам | `O(V)` | одно чтение `O(E)` | реализовано |
| Рассылка и сбор обновлений | меньше `O(V)` | запись и чтение `O(E)` обновлений | больше операций ввода и вывода |
| Двумерное разбиение | зависит от размера блока | чтение рёбер и повторное чтение рангов | сложнее подготовка и планирование |

Для проверенных графов массивы вершин помещаются в 128 МиБ, поэтому первый вариант оказался самым простым и быстрым
