# Способ хранения и альтернативы

## Реализованный способ

Использовано разбиение по конечным вершинам со сбором входящих вкладов

В литературе этот способ называется destination-sharded pull

После внешней сортировки каждое ребро хранится как пара `(destination, source)`

Файлы владеют непересекающимися диапазонами конечных вершин, поэтому два потока не записывают ранг одной вершины одновременно

В оперативной памяти находятся два массива рангов, исходящие степени и небольшие буферы чтения

Сам список рёбер остаётся на диске

### Подготовка графа

- первый проход создаёт временные отсортированные серии идентификаторов
- серии сливаются, а вершины получают последовательные внутренние номера
- второй проход преобразует рёбра во внутренний формат
- части рёбер сортируются несколькими потоками
- итоговое слияние удаляет повторы, считает исходящие степени и создаёт файлы графа
- число рёбер и контрольная сумма сверяются между проходами

Готовый каталог появляется только после успешного завершения всех этапов

### Причины выбора

- каждое ребро читается один раз за итерацию
- чтение идёт последовательно
- не создаются временные обновления для каждого ребра
- в основном цикле не нужны блокировки и атомарные сложения
- степень отдельной вершины не влияет на расход памяти
- порядок сложения не зависит от расписания потоков

Недостаток этого решения в том, что массивы состояния вершин должны помещаться в оперативную память

## Рассылка и сбор обновлений

Этот способ также известен как scatter/gather

Рёбра сначала обрабатываются по исходным вершинам, а вклад каждого ребра записывается во временный файл нужной части графа

После этого временные файлы читаются и суммируются по конечным вершинам

Так можно держать ранги на диске частями и обрабатывать граф с очень большим числом вершин

Цена такого решения — запись и повторное чтение `O(E)` обновлений на каждой итерации

Для PageRank это заметно увеличивает нагрузку на накопитель

## Двумерное разбиение матрицы

Матрица смежности делится на блоки по исходным и конечным вершинам

В памяти одновременно находятся часть исходных рангов и накопитель для части конечных вершин

Размер блока можно подобрать под жёсткое ограничение памяти

При этом исходные ранги приходится читать повторно для разных частей матрицы, а подготовка и расписание вычислений становятся сложнее

## Краткое сравнение

| Способ | Память | Работа с диском за итерацию | Реализация |
|---|---:|---:|---:|
| Разбиение по конечным вершинам | `O(V)` | одно чтение `O(E)` | реализовано |
| Рассылка и сбор обновлений | меньше `O(V)` | запись и чтение `O(E)` | сложнее |
| Двумерное разбиение | задаётся размером блока | чтение рёбер и повторное чтение рангов | сложнее |

Для проверенных графов состояние вершин помещается в 128 МиБ, поэтому дополнительная запись обновлений на диск не нужна
