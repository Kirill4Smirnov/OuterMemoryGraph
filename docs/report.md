# Отчёт: PageRank для ориентированного графа во внешней памяти

## 1. Цель

Требуется вычислить глобальную аналитическую метрику ориентированного
невзвешенного графа на одном Linux x86-64 узле, когда структура графа не
помещается в доступную оперативную память. Целевой пример ограничения — 128 MiB
RAM при графе размером от нескольких сотен мегабайт до гигабайта.

Дополнительные требования:

- использовать SSD эффективно;
- обрабатывать вершины с сильно неравномерной степенью;
- задействовать несколько CPU-ядер;
- получать детерминированный результат;
- предоставить воспроизводимые инструкции запуска и проверку корректности.

## 2. Выбранная метрика

Выбран **PageRank**. Он назначает каждой вершине вероятность нахождения
случайного пользователя в этой вершине после большого числа переходов по
исходящим рёбрам с периодической телепортацией.

Для вершины $v$:

$$
r_{t+1}(v) =
\frac{1-d}{N}
+ d \sum_{u \to v}\frac{r_t(u)}{\deg^+(u)}
+ \frac{d}{N}\sum_{w:\deg^+(w)=0}r_t(w),
$$

где:

- $N=|V|$;
- $d$ — damping factor, по умолчанию 0.85;
- $\deg^+(u)$ — исходящая степень;
- последнее слагаемое равномерно перераспределяет массу sink/dangling вершин.

Начальное распределение равномерное:

$$
r_0(v)=\frac{1}{N}.
$$

### 2.1. Почему PageRank

- Метрика применяется ко всему ориентированному графу.
- Результат имеет линейный размер $O(V)$.
- Одна итерация является разреженным matrix-vector multiplication и требует
  $O(V+E)$ операций.
- Алгоритм допускает последовательное потоковое чтение рёбер.
- Семантика dangling vertices и эталонные результаты стандартизованы в LDBC
  Graphalytics.
- При $d<1$ оператор PageRank является сжимающим в $L_1$, поэтому существует
  единственное стационарное распределение.

Грубая оценка ошибки:

$$
\lVert r_k-r^*\rVert_1 \le 2d^k.
$$

Для $d=0.85$ и ошибки $10^{-8}$ эта универсальная верхняя оценка даёт около
118 итераций. Реальное число зависит от структуры графа.

### 2.2. Рассмотренные альтернативы

1. **Katz centrality.** Также сводится к итеративному SpMV, но требует выбирать
   $\alpha < 1/\rho(A)$. Оценка спектрального радиуса добавляет отдельную
   задачу и новые проходы по графу.
2. **LeaderRank.** Не требует damping factor, но добавляет ground node, связанный
   со всеми вершинами. Он искусственно создаёт гипер-узел степени $N$ и менее
   удобен для независимой валидации.
3. **Jaccard/Dice.** Это метрики пар вершин. Вычисление для всех пар может
   породить $O(V^2)$ результатов, а гипер-узлы приводят к большому числу
   пересечений списков соседей.

PageRank имеет наиболее ясное сочетание полезности, вычислительной сложности и
проверяемости.

## 3. Варианты внешнепамятной архитектуры

### 3.1. Destination-sharded semi-external pull — реализован

Рёбра сортируются по `(destination, source)` и делятся на файлы-шарды. Каждый
шард владеет непересекающимся диапазоном destination-вершин.

На итерации:

1. По массивам рангов и степеней вычисляется dangling mass.
2. Для всех вершин устанавливается базовый teleport/dangling вклад.
3. Рабочие потоки получают независимые шарды.
4. Рёбра каждого шарда читаются последовательно.
5. Поток добавляет вклад source в `next_rank[destination]`.
6. Для диапазона шарда вычисляются локальные residual и rank sum.
7. Локальные значения сводятся главным потоком в порядке номеров шардов.

В RAM находятся два массива `double` и массив `uint32` степеней:

$$
M_{PR} \approx 8V+8V+4V = 20V \text{ байт}.
$$

Преимущества:

- одно чтение каждого ребра за итерацию;
- нет edge-sized временных updates;
- нет атомиков и блокировок в горячем цикле;
- порядок сложения для вершины фиксирован;
- adjacency list любой длины обрабатывается потоково.

Ограничение: состояние $O(V)$ должно помещаться в RAM. Сам граф $O(E)$
остаётся на SSD и может быть значительно больше памяти.

### 3.2. Edge-centric scatter/gather

Source-партиция загружается в память, после чего вклад каждого ребра записывается
во временный файл destination-партиции. Затем destination-партиция агрегирует
свои updates.

Преимущество — даже состояние $O(V)$ может храниться на диске. Недостаток —
на каждой итерации возникает запись и повторное чтение $O(E)$ update-записей.
Для PageRank это существенно увеличивает SSD traffic.

### 3.3. Двумерные source/destination tiles

Матрица смежности делится на блоки `(source_partition, destination_partition)`.
В памяти находится один destination accumulator и один source state block.

Подход строго ограничивает память без update-файла на каждое ребро, но требует
повторно читать source-состояние для разных destination-партиций и значительно
усложняет preprocessing и планирование.

Для заданного масштаба выбран первый вариант: он выполняет меньше I/O и проще
обеспечивает детерминированную многопоточность.

## 4. Предварительная обработка

Исходный CSV/SNAP edge-list читается два раза. Весь файл никогда не загружается
в память.

### 4.1. Фаза vertex runs

- В память набирается ограниченная порция endpoint ID.
- ID сортируются и локально дедуплицируются.
- Отсортированная порция записывается как binary run.
- Runs сливаются k-way merge в `vertices.bin`.

Fan-in каждого merge ограничен 64 файлами. Если runs больше, выполняются
промежуточные merge-проходы с удалением уже слитых временных файлов. Поэтому
число открытых файлов и суммарный объём входных буферов не растут вместе с
размером графа.

Исходные signed int32 ID сортируются. Их позиция в `vertices.bin` становится
плотным внутренним `uint32` ID.

### 4.2. ID mapping

Если диапазон исходных ID достаточно компактен, строится direct array mapping:

$$
dense = map[original-min].
$$

Это используется для обоих выбранных датасетов:

- `cit-Patents`: 3,774,768 вершин в диапазоне длиной 6,009,554;
- `soc-LiveJournal1`: плотный диапазон длиной 4,847,571.

Если диапазон слишком разреженный, используется бинарный поиск в отсортированном
`vertices.bin`. Этот fallback медленнее, но сохраняет корректность для любого
signed int32 ID и не требует неограниченной hash table.

### 4.3. Фаза edge runs

- Исходный edge-list читается повторно.
- Оба endpoint переводятся в dense ID.
- Ребро представляется как `(destination, source)` из двух `uint32`.
- Ограниченная порция сортируется и локально дедуплицируется.
- Порция сохраняется в binary edge run.

### 4.4. Финальный merge

Edge runs сливаются priority queue. Во время финального merge:

- удаляются дубли между runs;
- считаются исходящие степени;
- считаются self-loops;
- формируются edge-balanced destination shards;
- записывается manifest.

Перед финальным merge то же bounded-fan-in правило сводит любое число edge runs
не более чем к 64. Дополнительные проходы требуются только для входов существенно
крупнее протестированных датасетов.

Manifest публикуется последним. Весь граф сначала создаётся в staging directory,
валидируется, после чего directory атомарно переименовывается. Ошибка или
прерывание не оставляют частичный каталог, выглядящий валидным.

### 4.5. Сложность preprocessing

CPU:

$$
O(E\log E)
$$

в терминах общей внешней сортировки. I/O-сложность:

$$
O\left(\frac{E}{B}\log_{M/B}\frac{E}{B}\right),
$$

где $M$ — доступная память, $B$ — размер дискового блока.

Для run generation используется не более 80% заявленного memory budget; остаток
оставлен для процесса, allocator metadata, потоковых объектов и сортировки.

## 5. Внутренний формат

Version 1 ориентирован на little-endian x86-64:

- `vertices.bin`: массив исходных `int32` ID в dense-порядке;
- `outdegrees.bin`: массив `uint32` исходящих степеней;
- `shard-NNNNN.bin`: восьмибайтовые `(uint32 destination, uint32 source)`;
- `manifest.json`: версия, размеры, семантика и диапазоны шардов.

Шарды имеют непересекающиеся непрерывные vertex ranges и вместе покрывают
`[0, V)`. Размеры всех файлов сверяются с manifest перед PageRank.

Подробная спецификация находится в [file-format.md](file-format.md).

## 6. Многопоточность

Целевая машина имеет 8 физических ядер, 16 hardware threads и один NUMA node.
Это упрощает размещение памяти: отдельная NUMA-aware репликация не нужна.

Шарды выдаются потокам через атомарный счётчик заданий. Для одного шарда:

- только один поток читает его edge file;
- только этот поток пишет соответствующий диапазон `next_rank`;
- `rank` и `outdegree` доступны только для чтения.

Поэтому в горячем цикле нет:

- `mutex`;
- atomic floating-point additions;
- private rank vector на поток;
- merge конфликтующих destination updates.

Каждый шард сохраняет local residual и rank sum. Главный поток складывает их в
фиксированном порядке с compensated summation. Планирование потоков не меняет
порядок сложения рёбер одной вершины.

Автоматический тест запускает один граф с 1 и 4 потоками и требует побайтово
одинаковый CSV.

## 7. Гипер-узлы

Adjacency list не загружается целиком. Файл читается блоками по 128 KiB, поэтому
память не зависит от степени отдельной вершины.

В тестах создаётся вершина с:

- 50,000 исходящих рёбер;
- 50,000 входящих рёбер.

Preprocessing и PageRank проходят при memory budget 32 MiB.

Текущее ограничение: все входящие рёбра одной destination-вершины принадлежат
одному шарду, чтобы не требовать atomics. Если степень одной вершины сравнима со
всем $E$, такой шард остаётся memory-safe, но становится parallel straggler.
Возможное развитие — специальные hyper-node chunks с фиксированным порядком
редукции partial sums.

## 8. Корректность и воспроизводимость

### 8.1. Инварианты

После каждой итерации вычисляются:

- $\lVert r_{t+1}-r_t\rVert_1$;
- $\sum_v r_{t+1}(v)$.

Перед сохранением результата проверяется:

$$
\left|\sum_v r(v)-1\right| \le 10^{-10}.
$$

NaN/Inf нарушают эту проверку и завершают программу ошибкой.

### 8.2. Dangling graph

Для графа `1 -> 2` при $d=0.85$ аналитическое решение:

$$
r(1)=0.3508771929824561,
\qquad
r(2)=0.6491228070175439.
$$

Автоматический тест сравнивает результат с этими значениями с абсолютной
погрешностью $10^{-12}$.

### 8.3. LDBC Graphalytics

В репозитории находится официальный `test-pr-directed`:

- 50 вершин;
- 246 рёбер;
- $d=0.85$;
- 14 итераций;
- опубликованный reference output.

Graphalytics допускает относительную ошибку $10^{-4}$. Реализованный алгоритм
получил:

```text
maximum relative error = 1.2677113294486482e-06
maximum absolute error = 2.72355405381286e-08
```

### 8.4. Sanitizers и static analysis

- Все тесты проходят в Debug и Release-конфигурациях.
- Полный набор тестов проходит под AddressSanitizer и UndefinedBehaviorSanitizer.
- `clang-tidy` обрабатывает все production translation units без диагностик.

### 9.3. Preprocessing

| Dataset | Wall time | Max RSS | Vertex runs | Edge runs | Shard edges min/max |
|---|---:|---:|---:|---:|---:|
| cit-Patents | 10.58 s | 108,012 KiB | 2 | 2 | 257,620 / 258,196 |
| soc-LiveJournal1 | 43.80 s | 108,344 KiB | 6 | 7 | 1,072,507 / 1,080,699 |

В обоих наборах дубли отсутствовали. Обнаружены:

- 1 self-loop в `cit-Patents`;
- 518,382 self-loops в `soc-LiveJournal1`.

### 9.4. PageRank

Параметры: 16 потоков, $d=0.85$, global L1 tolerance $10^{-8}$.

| Dataset | Iterations | Final residual | Wall time including CSV | Max RSS |
|---|---:|---:|---:|---:|
| cit-Patents | 20 | 5.8122e-9 | 1.13 s | 80,784 KiB |
| soc-LiveJournal1 | 78 | 9.0892e-9 | 22.33 s | 102,280 KiB |

Измерения выполнялись с тёплым filesystem cache. Поэтому они характеризуют
реализацию и многопоточную обработку, но не являются честным сравнением
пропускной способности разных SSD.

### 9.5. Жёсткий cgroup limit

LiveJournal был запущен в user systemd cgroup:

```text
MemoryMax=128M
MemorySwapMax=0
threads=16
```

Две полные итерации и CSV-вывод завершились успешно:

```text
Memory peak: 127.9M
swap: 0B
```

Это значение включает не только anonymous RSS, но и file-cache pages,
начисленные cgroup. Близость к лимиту ожидаема: ядро использует свободный остаток
для read-ahead и вытесняет cache pages под memory pressure.

### 9.6. Масштабирование по потокам

Для `cit-Patents` выполнено по два warm-cache запуска на каждое число потоков,
по 5 итераций в запуске. В таблице приведена медиана всех десяти iteration
measurements для каждого значения. CSV создавался в каждом запуске, но время
итерации берётся из внутреннего таймера и не включает сериализацию результата.

| Threads | Median iteration | Median edges/s | Speedup |
|---:|---:|---:|---:|
| 1 | 0.2932 s | 56.34 M | 1.00x |
| 2 | 0.1443 s | 114.45 M | 2.03x |
| 4 | 0.0751 s | 220.07 M | 3.91x |
| 8 | 0.0461 s | 358.29 M | 6.36x |
| 16 | 0.0400 s | 412.55 M | 7.32x |

После 8 потоков масштабирование ожидаемо замедляется: используются SMT threads,
а потоки совместно потребляют memory bandwidth и filesystem cache. Отдельные
измерения содержали заметный системный шум, поэтому в итоговом benchmark следует
увеличить число повторов и фиксировать cold/warm-cache protocol.

## 10. Ограничения

1. Состояние $20V$ должно помещаться в память. Для графов с десятками
   миллионов вершин нужен строгий partitioned state вариант.
2. Version 1 поддерживает только little-endian x86-64.
3. Граф статичен после preprocessing.
4. Edge-only input не может описать изолированные вершины.
5. Preprocessing run generation пока однопоточный; PageRank многопоточный.
6. Очень крупный единственный destination hyper-node может ограничить
   масштабирование, хотя не нарушает memory bound.
7. Внутренние edge files не сжаты. Это увеличивает I/O, но уменьшает CPU cost и
   упрощает фиксированный memory budget.
8. Текущий convergence criterion — разность соседних итераций. Для оценки
   ошибки относительно fixed point можно использовать
   $\lVert r_{t+1}-r_t\rVert_1/(1-d)$.

## 11. Возможные улучшения

- параллельная bounded-memory генерация sorted runs;
- отдельные partial-sum chunks для экстремальных destination hyper-nodes;
- строгий external-state режим с 2D tiles;
- block compression с независимым индексом блоков;
- checkpoints rank vectors между итерациями;
- optional `vertices.csv` для isolated vertices;
- cold-cache benchmark protocol и thread-scaling table;
- prefetch/reciprocal-degree fast path при наличии дополнительной памяти.

## 12. Вывод

Destination-sharded pull representation позволяет превратить случайные обходы
графа в параллельные последовательные проходы по SSD. Для PageRank этот подход
выгоднее scatter/gather, потому что не записывает обновление на диск для каждого
ребра на каждой итерации.

Реализация обработала 1.08 ГБ исходного LiveJournal edge-list и 68.99 млн рёбер
на одном узле, прошла официальный Graphalytics reference, тест гипер-узла,
sanitizers и жёсткий cgroup limit 128 MiB без swap.
