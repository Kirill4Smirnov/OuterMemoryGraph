# Отчёт по решению

## Задача

Нужно вычислить аналитическую метрику ориентированного графа на одном компьютере, когда список рёбер не помещается в оперативную память

Для проверки принят предел 128 МиБ и твердотельный накопитель

Решение должно использовать несколько ядер, не зависеть по памяти от степени отдельной вершины и давать воспроизводимый результат

## Выбор метрики

В проекте вычисляется PageRank

Он оценивает важность вершины с учётом входящих ссылок и важности их источников

Такой подход используют для ранжирования страниц, пользователей, публикаций и других объектов со связями

PageRank хорошо подходит для задания

- метрика вычисляется для всего ориентированного графа
- одна итерация требует линейного прохода по вершинам и рёбрам
- итог содержит одно число на вершину
- для проверки есть открытый эталон LDBC Graphalytics

Для вершины $v$ используется формула

$$
r_{t+1}(v) =
\frac{1-d}{N}
+ d \sum_{u \to v}\frac{r_t(u)}{\deg^+(u)}
+ \frac{d}{N}\sum_{w:\deg^+(w)=0}r_t(w)
$$

Здесь $N$ — число вершин, $d$ — коэффициент затухания, а $\deg^+(u)$ — исходящая степень

Последнее слагаемое возвращает ранг вершин без исходящих рёбер обратно в граф

Начальное значение для каждой вершины равно $1/N$

## Рассмотренные способы обработки

Были рассмотрены три способа

1. Разбиение рёбер по конечным вершинам со сбором входящих вкладов
2. Рассылка обновлений через временные файлы и их последующая сборка
3. Двумерное разбиение матрицы смежности на блоки

Реализован первый способ

Он требует хранить состояние всех вершин в памяти, зато за итерацию читает каждое ребро один раз и ничего не записывает на диск

Второй и третий способы позволяют обрабатывать состояние вершин частями, но требуют больше операций с накопителем и заметно усложняют код

Подробное сравнение есть в [отдельном документе](alternatives.md)

## Подготовка графа

Текстовый список рёбер неудобно разбирать на каждой итерации, поэтому сначала строится двоичный граф

Подготовка идёт в два прохода

### Первый проход

- идентификаторы набираются небольшими порциями
- каждая порция сортируется и записывается во временный файл
- временные файлы сливаются
- вершины получают внутренние номера от `0` до `V - 1`

Исходные идентификаторы могут быть отрицательными и должны помещаться в `int32`

### Второй проход

- исходные идентификаторы заменяются внутренними номерами
- ребро разворачивается в пару `(конечная вершина, исходная вершина)`
- ограниченные порции рёбер сортируются несколькими потоками
- итоговое слияние удаляет повторы и считает исходящие степени
- рёбра делятся на файлы с непересекающимися диапазонами конечных вершин

Между проходами сверяются число рёбер и контрольная сумма, поэтому изменение входного файла не останется незамеченным

Готовый каталог публикуется только после успешного завершения всех этапов

## Итерация PageRank

На каждой итерации программа

1. Считает сумму рангов вершин без исходящих рёбер
2. Заполняет новый массив базовым вкладом
3. Раздаёт файлы графа рабочим потокам
4. Последовательно читает рёбра и собирает входящие вклады
5. Считает L1-разность соседних векторов и сумму рангов

Каждый файл отвечает за отдельный диапазон вершин

Два потока не записывают один элемент массива, поэтому в основном цикле нет блокировок и атомарных сложений

Файлы назначаются потокам динамически, что помогает при разном размере частей

Сложение рёбер одной вершины всегда идёт в одном порядке

## Работа с гиперузлами

Список соседей вершины не собирается в память

Даже если у вершины десятки тысяч рёбер, они читаются по одному из последовательного файла

Это сохраняет ограничение памяти, но одна вершина с огромной входящей степенью может задержать поток, которому достался её файл

Разделить такую вершину между потоками можно только с дополнительным этапом детерминированного сложения частичных сумм

## Память и сложность

Во время PageRank в памяти находятся

- текущие ранги `8 * V` байт
- новые ранги `8 * V` байт
- исходящие степени `4 * V` байт
- небольшие буферы потоков

Основная оценка равна примерно `20 * V` байт

Список рёбер в неё не входит и может быть намного больше доступной памяти

Пусть $E$ — число уникальных рёбер, а $K$ — число итераций

- подготовка требует до `O(E log E)` сравнений
- одна итерация требует `O(V + E)` операций
- полный расчёт требует `O(K(V + E))` операций
- за итерацию с диска читается `O(E)` данных

Остановка возможна после заданного числа итераций или при достижении нужной L1-разности

L1-разность соседних векторов не равна точной ошибке относительно стационарного решения

Для $d < 1$ можно использовать оценку

$$
\lVert r_t-r^*\rVert_1
\leq
\frac{\lVert r_t-r_{t-1}\rVert_1}{1-d}
$$

## Проверка корректности

В проекте есть 12 автоматических тестов

Среди них проверка вершин без исходящих рёбер, повторных рёбер, петель, гиперузла и одинакового результата при разном числе потоков

На эталоне LDBC Graphalytics максимальная относительная ошибка составила `1.2677113296e-06` при допустимом значении `1e-4`

Перед записью результата программа проверяет конечность рангов и близость их суммы к единице

Подробности собраны в [описании проверок](verification.md)

## Проверка на больших графах

Стенд

- AMD Ryzen 7 8845H
- 8 ядер и 16 аппаратных потоков
- NVMe SSD
- оптимизированная сборка GCC

Использовались два набора SNAP

| Набор | Вершины | Уникальные рёбра | Текстовый файл |
|---|---:|---:|---:|
| `cit-Patents` | 3 774 768 | 16 518 948 | 280 538 727 байт |
| `soc-LiveJournal1` | 4 847 571 | 68 993 773 | 1 080 598 042 байта |

Подготовка с пределом 128 МиБ и 64 частями

| Набор | Время | Наибольший RSS |
|---|---:|---:|
| `cit-Patents` | 10.58 с | 108 012 КиБ |
| `soc-LiveJournal1` | 43.80 с | 108 344 КиБ |

PageRank с 16 потоками, $d = 0.85$ и L1-разностью не больше $10^{-8}$

| Набор | Итерации | Итоговая L1-разность | Время вместе с CSV | Наибольший RSS |
|---|---:|---:|---:|---:|
| `cit-Patents` | 20 | `5.8122e-9` | 1.13 с | 80 784 КиБ |

Эти замеры сделаны с тёплым файловым кешем и не подходят для сравнения разных накопителей

Отдельный запуск `soc-LiveJournal1` на две итерации завершился в контрольной группе с пределом 128 МиБ и отключённым файлом подкачки

Пиковое значение для всей контрольной группы вместе с файловым кешем составило `127.9 МиБ`

## Ограничения

- массивы состояния размером около `20 * V` байт должны помещаться в память
- из списка рёбер нельзя узнать об изолированных вершинах
- внутренний формат рассчитан на Linux x64 с порядком байтов от младшего к старшему
- подготовленный граф нельзя изменять
- одна конечная вершина обрабатывается одним потоком
- промежуточные ранги не сохраняются для продолжения после сбоя
- рёбра во внутреннем формате не сжимаются

## Что можно улучшить

- добавить отдельный список вершин
- делить гиперузлы на части с фиксированным порядком суммирования
- сохранять состояние между итерациями
- сжимать блоки рёбер
- добавить обработку рангов частями для графов, где `20 * V` байт не помещаются в память

## Вывод

Выбранный способ оставляет рёбра на накопителе и читает их последовательно

Он не зависит по памяти от степени вершины, использует несколько ядер и даёт одинаковый результат при повторных запусках

Проверка на `soc-LiveJournal1` показала, что граф с текстовым файлом больше 1 ГБ обрабатывается при пределе 128 МиБ
