## Краткий вывод

Для этого задания рекомендуется выбрать **PageRank** и реализовать его на **C++20** как semi-external алгоритм:

- рёбра хранятся на SSD в бинарных отсортированных шардах;
- в оперативной памяти остаются только два массива рангов и степени исхода вершин;
- итерации выполняются преимущественно последовательным чтением диска;
- независимые диапазоны принимающих вершин обрабатываются параллельно;
- результат остаётся детерминированным и проверяемым.

Такое решение действительно обрабатывает граф, не помещающийся в память, не маскирует задачу готовым графовым фреймворком, естественно распараллеливается и имеет реалистичный объём разработки.

## Что на самом деле проверяет задание

Это не просто задача «написать PageRank». Основная сложность заключается в организации вычислений так, чтобы алгоритм почти всё время выполнял **последовательное чтение SSD**, а не случайно обращался к разным местам огромного файла.

Главные инженерные проблемы:

- CSV слишком медленный и объёмный для итерационной обработки. Он должен быть только входным и выходным форматом.
- Обычный список смежности невозможно полностью держать в RAM.
- Случайный доступ к рёбрам на диске резко снижает производительность даже на SSD.
- Гипер-узел нельзя загружать в отдельный `vector` целиком: его рёбра нужно обрабатывать потоково.
- Многопоточное сложение `double` через атомики может дать недетерминированный результат.
- Ограничение 128 МБ должно соблюдаться реально, а не только «по расчётам».
- Долгая предварительная обработка допустима, если после неё каждая итерация выполняется эффективно.

Просто применить `mmap` к огромному графу недостаточно. В худшем случае это передаст управление памятью ядру Linux и приведёт к постоянным page fault и вытеснениям страниц.

## Выбор метрики

| Метрика | Преимущества | Проблемы для задания | Вывод |
|---|---|---|---|
| PageRank | Глобальная, ориентированная, известная, детерминированная, хорошо валидируется | Итеративная; нужно правильно обрабатывать висячие вершины | Лучший выбор |
| LeaderRank | Нет damping-параметра, подходит ориентированным графам | Добавляет ground node степени $N$, менее стандартные эталоны | Можно использовать, но сложнее защищать |
| Katz centrality | Глобальная центральность, сводится к разреженному matrix-vector multiplication | Нужно выбирать $\alpha < 1/\rho(A)$, иначе возможна расходимость | Добавляет лишний спектральный вопрос |
| HITS | Выдаёт authority и hub score | Требует двух связанных векторов, чувствителен к структуре компонент | Допустимо, но PageRank проще |
| Jaccard/Dice | Полезны для link prediction | Для всех пар результат может иметь размер $O(V^2)$; гипер-узлы создают взрыв пересечений | Не рекомендуется |
| Betweenness | Интересная глобальная центральность | Точный алгоритм слишком дорог: порядка $O(VE)$ | Не подходит |

### Почему PageRank

PageRank отвечает на понятный вопрос: насколько важна вершина с учётом важности вершин, которые на неё ссылаются.

Преимущества для проекта:

- алгоритм проходит по всему графу;
- ориентированность используется естественным образом;
- каждая итерация имеет линейную сложность;
- результат можно сравнить с NetworkX и Graphalytics;
- телепортация обеспечивает единственное стационарное распределение;
- алгоритм хорошо сводится к последовательному проходу по рёбрам.

### Почему не LeaderRank

LeaderRank интересен тем, что не требует damping factor, но он искусственно добавляет ground node, соединённый со всеми вершинами. Этот узел сразу становится гипер-узлом степени $N$. Его можно обрабатывать аналитически, не материализуя рёбра, но это усложняет объяснение, реализацию и валидацию.

### Почему не Katz

Katz centrality имеет вид:

$$
x = \alpha A x + \beta \mathbf{1}
$$

Для сходимости необходимо выбрать:

$$
\alpha < \frac{1}{\rho(A)},
$$

где $\rho(A)$ — спектральный радиус матрицы смежности. Это создаёт дополнительную задачу оценки спектрального радиуса и объяснения выбора параметра.

### Почему не Jaccard и Dice

Jaccard и Dice естественно являются метриками сходства пар вершин. Если считать их для всех пар, число потенциальных результатов достигает $O(V^2)$. Даже если считать только для существующих рёбер или кандидатов, гипер-узлы создают большое число пересечений множеств соседей.

## Как работает PageRank

Для вершины $v$:

$$
PR_{t+1}(v) =
\frac{1-d}{N}
+
d \sum_{u \rightarrow v}
\frac{PR_t(u)}{\operatorname{outdeg}(u)}
+
\frac{d}{N}\sum_{w:\operatorname{outdeg}(w)=0}PR_t(w)
$$

Обычно используется:

$$
d = 0.85.
$$

Последнее слагаемое принципиально важно: это перераспределение ранга dangling vertices — вершин без исходящих рёбер. Если его забыть, сумма рангов начнёт уменьшаться, а результат будет неверным.

### Сходимость

При $d < 1$ отображение PageRank является сжимающим в норме $L_1$: ошибка уменьшается не хуже чем в $d$ раз за итерацию.

Грубая верхняя оценка:

$$
\|r_k-r^*\|_1 \le 2d^k.
$$

Для $d=0.85$ и целевой ошибки $10^{-8}$ эта оценка даёт около 118 итераций. На реальных графах обычно требуется несколько десятков итераций, но фактическое значение следует измерить экспериментально.

Стоит поддержать два режима:

1. Фиксированное число итераций — для совместимости с Graphalytics.
2. Остановка по глобальному $L_1$-residual — для получения сошедшегося результата.

## Варианты хранения и обработки

В отчёте достаточно предложить три подхода и реализовать один из них.

### 1. Destination-sharded pull PageRank — рекомендуемый вариант

Рёбра преобразуются в бинарные пары:

```cpp
uint32_t source;
uint32_t destination;
```

Они сортируются по `(destination, source)` и разбиваются на шарды по диапазонам destination.

На каждой итерации:

1. Рассчитывается сумма рангов dangling vertices.
2. Несколько потоков параллельно читают независимые шарды.
3. Для каждой destination-вершины поток последовательно суммирует вклады входящих соседей.
4. Поток записывает результат в принадлежащую ему часть массива `next_rank`.
5. Массивы `rank` и `next_rank` меняются местами.

В RAM находятся:

- `rank`: $8V$ байт;
- `next_rank`: $8V$ байт;
- `outdegree`: $4V$ байт;
- небольшие файловые и рабочие буферы.

Итого:

$$
20V + O(T \cdot buffer).
$$

Для LiveJournal с 4 847 571 вершиной состояние занимает примерно 92.5 MiB. Оно может поместиться в лимит 128 MiB при аккуратном контроле буферов.

Преимущества:

- каждое ребро читается один раз за итерацию;
- не создаётся временный файл размером $E$ на каждой итерации;
- чтения почти полностью последовательные;
- нет гонок и атомиков для destination-вершин;
- результат можно сделать детерминированным;
- adjacency list гипер-узла не загружается целиком.

Ограничение: состояние размером $O(V)$ должно помещаться в RAM. Это стандартная **semi-external** модель. Условие задания требует, чтобы в память не помещался весь граф, но не запрещает хранить компактное состояние вершин.

### 2. External-memory scatter/gather в стиле X-Stream

Вершины и рёбра делятся на партиции.

На scatter-фазе:

- читается source-партиция;
- вычисляются вклады исходящих рёбер;
- вклады записываются во временные файлы destination-партиций.

На gather-фазе:

- одна destination-партиция загружается в RAM;
- её updates последовательно читаются и агрегируются;
- обновлённое состояние сохраняется на диск.

Преимущества:

- даже массивы рангов не обязаны помещаться в память;
- используются преимущественно последовательные дисковые операции;
- произвольные int32 ID и гипер-узлы не являются проблемой.

Главный недостаток — большой write amplification. Для каждого ребра на каждой итерации нужно как минимум:

- прочитать ребро;
- записать update;
- снова прочитать update.

Для десятков миллионов рёбер это создаёт сотни мегабайт или гигабайты временных данных за одну итерацию.

### 3. Двумерное разбиение матрицы смежности

Матрица графа делится на блоки:

```text
(source_partition, destination_partition)
```

При обработке destination-блока его аккумулятор находится в RAM, а source-блоки рангов последовательно подгружаются.

Преимущества:

- строго ограниченная память;
- рёбра не требуют update-файла на каждой итерации;
- подход естественно интерпретируется как блочное sparse matrix-vector multiplication.

Недостатки:

- source-состояние приходится перечитывать для разных destination-блоков;
- preprocessing и планирование сложнее;
- может появиться много файлов или сложный индекс единого контейнера;
- распараллеливание сложнее, чем в semi-external pull-варианте.

Этот вариант полезен как направление развития, если даже $O(V)$ состояние не помещается в память.

## Рекомендуемый внутренний формат

CSV следует один раз преобразовать во внутреннее представление:

```text
edges.csv
    ↓ streaming parser
внешняя сортировка и перенумерация ID
    ↓
vertices.bin       original int32 ID для каждого dense ID
outdegree.bin      uint32
shard-000.bin      рёбра, отсортированные по destination
shard-001.bin
...
manifest.json      версия формата, V, E, параметры, checksums
```

### Зачем нужна перенумерация

ID формально являются `int32`, но не обязаны быть плотными или неотрицательными. Например, могут одновременно существовать вершины:

```text
-2000000000
17
1900000000
```

Создавать массив по максимальному ID нельзя. Поэтому вершины нужно перенумеровать в dense ID:

```text
0 ... V-1
```

Исходный ID сохраняется в `vertices.bin`, чтобы итоговый CSV содержал оригинальные идентификаторы.

### Внешняя сортировка

Перенумерация и сортировка должны также соблюдать лимит памяти:

1. Собирать ID и рёбра порциями.
2. Сортировать каждую порцию в RAM.
3. Удалять локальные дубликаты.
4. Сохранять отсортированные runs на диск.
5. Выполнять k-way merge.
6. Создать отображение original ID → dense ID.
7. Повторно преобразовать и отсортировать рёбра.
8. Во время финального merge удалить повторяющиеся рёбра и посчитать `outdegree`.

I/O-сложность внешней сортировки:

$$
O(\operatorname{sort}(E)) =
O\left(
\frac{E}{B}
\log_{M/B}\frac{E}{B}
\right),
$$

где:

- $M$ — доступная оперативная память;
- $B$ — размер дискового блока.

После preprocessing одна итерация PageRank требует:

- CPU: $O(V+E)$;
- RAM: $O(V)$;
- последовательный I/O: $O(E/B)$.

Полная сложность для $K$ итераций:

$$
O(K(V+E)).
$$

Универсального содержательного «среднего случая» для произвольных графов нет без заданного вероятностного распределения графов. Поэтому в отчёте лучше дать строгую худшую оценку и отдельно экспериментальные результаты на реальных датасетах.

## Обработка гипер-узлов

Степень 50 000 сама по себе не является проблемой. Проблема появляется только в случае, если программа пытается загрузить весь adjacency list вершины в память.

В destination-sharded схеме:

- входящие рёбра гипер-узла лежат подряд;
- поток читает их блоками, например по 1–4 MiB;
- в памяти хранится только текущая сумма;
- исходящая степень source-вершины читается из массива `outdegree`.

Для патологического случая, когда одна вершина содержит существенную долю всех рёбер, её adjacency list можно разделить на несколько chunk-задач. Каждая задача вычисляет partial sum, после чего partial sums складываются строго по номеру chunk.

Это одновременно сохраняет:

- ограничение памяти;
- параллелизм;
- воспроизводимый порядок сложения.

## Многопоточность и детерминизм

Pull-модель удобнее scatter-модели, потому что каждый destination-шард может принадлежать только одному потоку.

Следствия:

- не нужны атомарные сложения в общий массив;
- не нужны блокировки на каждой вершине;
- разные потоки записывают разные диапазоны `next_rank`;
- порядок сложения входящих рёбер внутри вершины остаётся фиксированным.

Для глобальных величин каждый шард должен возвращать локальный результат:

- dangling sum;
- $L_1$-residual;
- сумма новых рангов;
- статистика времени и числа обработанных рёбер.

Главный поток складывает локальные результаты строго по номеру шарда. Для повышения точности можно использовать Neumaier или Kahan summation.

Так результат не зависит от порядка завершения рабочих потоков.

### Ожидания от масштабирования

PageRank выполняет мало вычислений на одно ребро, поэтому алгоритм часто становится I/O-bound. Один или несколько потоков могут полностью насытить SSD раньше, чем будут загружены все CPU.

В отчёте важно показать:

- корректный параллельный дизайн;
- ускорение preprocessing;
- рост throughput до насыщения SSD;
- точку, после которой дополнительные потоки перестают помогать;
- отсутствие падения корректности при изменении числа потоков.

### LDBC Graphalytics

Graphalytics особенно полезен для этого задания:

- PageRank входит в шесть официальных алгоритмов бенчмарка;
- есть маленькие тесты и reference outputs;
- есть реальные и синтетические графы разных размеров;
- спецификация явно определяет обработку dangling vertices;
- значения должны обрабатываться как IEEE-754 `double`;
- проверка PageRank допускает относительную ошибку $10^{-4}$;
- по состоянию на февраль 2026 года опубликованы vertex/edge-файлы в Parquet.

Для основной C++-программы лучше использовать текстовый edge-list из архива либо потоково конвертировать его в CSV. Подключать Apache Arrow в ядро только ради Parquet нецелесообразно.

### SNAP

SNAP предоставляет множество реальных графов в простом edge-list формате. Особенно удобны:

- `web-Google` для промежуточных тестов;
- `cit-Patents` для ориентированного графа цитирований;
- `soc-LiveJournal1` для большого социального графа с тяжёлым распределением степеней.

Перед распространением датасета вместе с репозиторием нужно проверить его лицензию. Обычно безопаснее предоставлять download script и checksum, а не коммитить данные в Git.

### LAW/WebGraph

LAW предоставляет очень крупные web-графы в сжатом формате WebGraph. Эти данные хорошо подходят для серьёзных внешнепамятных экспериментов, но требуют отдельного Java-конвертера или поддержки формата WebGraph.

Поэтому LAW разумно оставить для дополнительного эксперимента, а не использовать как первый датасет.

### LDBC SNB

LDBC Social Network Benchmark позволяет генерировать воспроизводимые синтетические социальные графы заданного масштаба. Однако это гетерогенный граф с несколькими типами вершин и отношений. Для первого этапа он добавит много преобразований, не относящихся непосредственно к external-memory PageRank.

## Рекомендуемый стек технологий

Основной стек:

- C++20;
- GCC 11+;
- CMake;

- Python и NetworkX только для reference validation на маленьких графах;
- Dockerfile на базе Ubuntu 22.04;
- GitHub Actions с Ubuntu 22.04.

### Почему C++

- язык рекомендован в задании;
- легко контролировать память и бинарные форматы;
- минимальный runtime overhead;
- удобно использовать системный файловый API;
- нет необходимости отдельно оправдывать выбор языка;
- можно обеспечить простой запуск на Ubuntu 22.04.

### Что не рекомендуется для ядра

- **Spark/GraphX** — большой runtime overhead и скрытое управление памятью;
- **Neo4j** — база данных, а не эффективный streaming analytics engine;
- **RocksDB/SQLite для каждого ребра** — случайные lookup и write amplification;
- **чистый `mmap` без собственного разбиения** — непрозрачное управление resident set;
- **Parquet как итерационный формат** — лишняя декомпрессия и сложный runtime;
- **готовый GraphChi/WebGraph как основная реализация** — задание требует реализовать алгоритм и работу с памятью в собственном коде.

GraphChi, X-Stream и WebGraph при этом полезны как литературная основа и источники архитектурных идей.

## Проверка корректности

Одного просмотра итогового CSV недостаточно.

### 1. Аналитические графы

Нужны тесты как минимум для:

- ориентированного цикла — все ранги одинаковы;
- звезды;
- графа с dangling vertices;
- несвязных компонент;
- self-loop;
- гипер-узла;
- вершины без входящих рёбер;
- графа, в котором почти все вершины являются sink.

### 2. Graphalytics `test-pr-directed`

Официальный тест задаёт:

- 50 вершин;
- 246 рёбер;
- $d=0.85$;
- ровно 14 итераций;
- эталонный output.

Результат следует сравнивать с официальным файлом с относительной ошибкой не более $10^{-4}$.

### 3. NetworkX

На небольших графах нужно сравнить:

- одинаковую обработку dangling nodes;
- $L_1$-ошибку;
- maximum absolute error;
- maximum relative error;
- сумму рангов.

NetworkX нужен только как reference implementation в тестах, но не в основном алгоритме.

### 4. Инварианты каждой итерации

Нужно проверять:

- все значения конечны;
- `rank >= 0`;
- сумма рангов близка к 1;
- отсутствуют NaN и Inf;
- residual не ведёт себя аномально.

Пример лога:

```text
iteration=23 residual_l1=7.31e-09 rank_sum=1.0000000000000002
```

### 5. Детерминизм

Желательные проверки:

- одинаковый SHA-256 результата при нескольких запусках;
- одинаковый результат при одинаковом числе итераций;
- одинаковый или численно эквивалентный результат при `--threads 1`, `2`, `4`, `8`;
- стабильный порядок строк в CSV.

## Пограничные случаи

### Изолированные вершины

В edge-only CSV невозможно представить вершину без рёбер. Поэтому стоит поддержать необязательный файл:

```text
vertices.csv
```

Если он не передан, множество вершин определяется как объединение всех `from` и `to`.

### Повторяющиеся рёбра

Поскольку задан обычный невзвешенный граф, разумно по умолчанию удалять дубликаты на стадии внешней сортировки.

Поведение нужно явно описать в README:

- default: duplicate edges are deduplicated;
- optional mode: сохранить multiedges, если это когда-нибудь понадобится.

### Self-loops

PageRank корректно обрабатывает self-loop. Можно оставить такие рёбра, если формат задания их явно не запрещает.

### Типы данных

- исходный ID: `int32_t`;
- плотный внутренний ID: `uint32_t`;
- число вершин: как минимум `uint64_t` в метаданных;
- число рёбер и файловые offsets: `uint64_t`;
- outdegree: `uint32_t`, так как в простом графе степень не превышает число вершин;
- rank: `double`.

### Padding бинарных структур

Не следует слепо записывать:

```cpp
file.write(reinterpret_cast<const char*>(&edge), sizeof(edge));
```

если формат зависит от layout компилятора. Лучше явно сериализовать два 32-битных значения и зафиксировать endianess в документации формата.

### Временное место

Для preprocessing понадобится больше места, чем занимает итоговый граф. Безопасная предварительная оценка — от двух до четырёх размеров входного файла в зависимости от числа runs и необходимости хранить исходный бинарный edge-list.

### Восстановление после ошибки

Неполный preprocessing не должен выглядеть как валидный граф. Нужно:

- писать файлы с временными именами;
- сохранять manifest последним;
- использовать atomic rename;
- проверять версию формата и размеры файлов при открытии.

## Измерение памяти и производительности

В отчёте желательно привести:

- время preprocessing;
- время одной итерации;
- полное время PageRank;
- число итераций;
- throughput в edges/s;
- максимальный RSS;
- размер внутреннего графа;
- максимальный объём временных файлов;
- результаты для 1, 2, 4 и 8 потоков;
- checksum результата.

### Измерение RSS

Минимальная проверка:

```bash
/usr/bin/time -v ./build/omg ...
```

В выводе нужно смотреть:

```text
Maximum resident set size
```

Для строгого лимита лучше использовать Docker или cgroup:

```bash
docker run \
  --memory=128m \
  --memory-swap=128m \
  ...
```

### Файловый cache

Второй запуск может быть быстрее первого не из-за алгоритма, а потому что данные остались в системном файловом cache.

В отчёте следует:

- разделить cold и warm runs;
- либо явно указать, что измерение было warm-cache;
- не сравнивать первый запуск одного варианта со вторым запуском другого;
- повторять измерения и приводить медиану и диапазон.

Повторение benchmark не вносит случайность в сам алгоритм: вход и результат остаются детерминированными, а несколько запусков нужны только для оценки системного шума.

### Возможный CLI

Предварительная обработка:

```bash
./build/omg preprocess \
  --input edges.csv \
  --graph-dir data/graph \
  --work-dir data/tmp \
  --memory-mb 128 \
  --threads 8 \
  --shards 32
```

PageRank с фиксированным числом итераций:

```bash
./build/omg pagerank \
  --graph-dir data/graph \
  --output pagerank.csv \
  --damping 0.85 \
  --iterations 20 \
  --memory-mb 128 \
  --threads 8
```

PageRank до сходимости:

```bash
./build/omg pagerank \
  --graph-dir data/graph \
  --output pagerank.csv \
  --damping 0.85 \
  --tolerance 1e-8 \
  --max-iterations 100 \
  --memory-mb 128 \
  --threads 8
```

Отдельный vertex-list в version 1 не поддерживается: множество вершин равно
объединению endpoint всех рёбер, поэтому изолированные вершины из edge-only
входа восстановить невозможно.
